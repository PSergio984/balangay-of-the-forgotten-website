# Phase 2.75: Wiki Polish & Performance Optimization - Context

**Gathered:** 2026-05-26
**Status:** Ready for planning

<domain>
## Phase Boundary

This phase addresses critical issues identified after the initial wiki deployment, specifically regarding image rendering in production, page loading speed, sticky footer behavior, and wiki article rendering/formatting.

</domain>

<decisions>
## Implementation Decisions

### Image Asset Storage
- **D-01:** Configure the Payload `Media` collection's `staticDir` to save directly to `public/media/` and update the seeder to put them there.
- **D-02:** Commit the seeded media assets under `public/media/` to Git. This guarantees the images are bundled directly within Next.js's public build and are immediately available on serverless/production deployments.

### Page Loading Performance
- **D-03:** Implement Next.js `generateStaticParams` (Static Site Generation) on dynamic routes `app/(site)/wiki/[category]/[slug]/page.tsx` and `app/(site)/wiki/[category]/page.tsx` to pre-generate these pages at build time.
- **D-04:** Set up Incremental Static Regeneration (ISR) with a revalidation limit (e.g. `revalidate = 3600`) to fetch fresh records if changes occur in CMS database.

### Sticky Footer shared layout
- **D-05:** Extract the footer from `app/(site)/page.tsx` and turn it into a shared `Footer` component.
- **D-06:** Place the `Footer` component inside the shared `app/(site)/layout.tsx`. Use a CSS flexbox layout (`flex flex-col min-h-screen` on the layout wrapper, `flex-grow` on the main page wrapper, and position the footer at the bottom) so it sticks cleanly on short pages.

### Wiki Layout & Formatting
- **D-07:** Replace the placeholder string (`Detailed lore description from archives...`) on the wiki entry page with a dedicated Lexical rich-text renderer that reads and renders the real JSON structure of `doc.description` into HTML elements.

### the agent's Discretion
- The exact layout style/spacing of the rich text parser (paragraphs, lists, and headers formatting) is at the agent's discretion, as long as it matches the pixel-art/scroll aesthetic.

</decisions>
