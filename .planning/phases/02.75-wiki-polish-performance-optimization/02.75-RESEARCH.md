## 1. Image Loading on Deployment

### Current State
- **Media collection** (`collections/Media.ts` L3-16): Uses `upload: true` with no `staticDir` or `staticURL` configured. Payload defaults to saving uploads to `<project-root>/media/`.
- **Actual upload location**: `media/` directory at project root contains 13 PNG files (characters, bosses, relics). These are NOT in `public/` â€” Next.js cannot serve them in production.
- **Seed script** (`scripts/seed-wiki-full.ts` L27-52): Uses `payload.create()` with a file buffer to upload images. Payload stores them wherever `staticDir` points (currently the default `media/`).
- **`.gitignore`**: No rule excludes `media/` or `public/media/`. Both directories would be committed if they exist.
- **Image rendering** in wiki pages (`app/(site)/wiki/[category]/[slug]/page.tsx` L126-132): Uses `<Image src={doc.image.url} />`. The `url` field on Media records returns paths like `/api/media/file/Bathala.png` by default (Payload's built-in route). This works in dev (Payload serves it), but in production/static builds the API route may not be available or may be slow.

### Root Cause of Broken Images in Production
Payload's default `upload: true` stores files to `<root>/media/` and serves them through its own API handler (`/api/media/file/<filename>`). In serverless deployments:
1. The filesystem is ephemeral â€” files uploaded at seed time don't persist between deployments.
2. The `media/` directory is outside `public/`, so Next.js static hosting doesn't serve them.

### Recommended Fix (Per Decision D-01, D-02)
1. **Update `collections/Media.ts`** to set explicit upload config:
   ```ts
   upload: {
     staticDir: 'public/media',
     staticURL: '/media',
   }
   ```
2. **Move existing files** from `media/` to `public/media/`.
3. **Update the seed script** â€” no code changes needed since `payload.create()` with file buffer respects the collection's `staticDir`.
4. **Re-run the seeder** after config change so that DB records have correct URLs (`/media/Bathala.png` instead of `/api/media/file/Bathala.png`).
5. **Commit `public/media/`** to git â€” since `.gitignore` has no rule excluding it, images will be bundled in the build.
6. **Update `next.config.mjs`** â€” add image domain/pattern config if using `next/image`:
   ```js
   const nextConfig = {
     images: {
       remotePatterns: [], // Not needed since images are local
     },
   }
   ```
   No remote patterns needed â€” local `public/` files work automatically with `next/image`.

### Risk
- After changing `staticDir`, existing DB records still reference old URLs. Must re-seed or run a migration to update `url` fields.
- The old `media/` directory should be deleted after migration to avoid confusion.

## 2. Page Loading Performance (SSG/ISR)

### Current State
- **Wiki hub** (`app/(site)/wiki/page.tsx`): Pure static â€” no data fetching, hardcoded categories. Already fast.
- **Category page** (`app/(site)/wiki/[category]/page.tsx` L32-35): Server-side `payload.find()` with `limit: 100`. No `generateStaticParams`, no `revalidate`. Rendered on every request (SSR).
- **Entry page** (`app/(site)/wiki/[category]/[slug]/page.tsx` L32-40): Server-side `payload.find()` with `where: { slug }`, `limit: 1`. No `generateStaticParams`, no `revalidate`. Also SSR on every request.
- **Homepage** (`app/(site)/page.tsx` L14-18): Fetches news with `payload.find()`. No caching/ISR.
- **No `depth` parameter** is passed to any `payload.find()` call â€” Payload defaults to `depth: 1`, which auto-populates relationship fields one level deep. This is fine for current needs but could be optimized.
- **Next.js version**: `16.2.6` â€” supports `generateStaticParams` and `revalidate` config exports.
- **Payload version**: `3.84.1`.

### Query Optimization Opportunities
- Category page fetches ALL fields with `limit: 100`. Should add `select` to only fetch `name`, `slug`, `image` for listing cards.
- Entry page queries by `slug` â€” optimal. Could add `depth: 1` explicitly to ensure relationships (location, relics) resolve.

### Recommended Approach (Per Decision D-03, D-04)

**Category page** â€” add `generateStaticParams`:
```tsx
export async function generateStaticParams() {
  return [
    { category: 'bosses' },
    { category: 'characters' },
    { category: 'relics' },
    { category: 'locations' },
    { category: 'minibosses' },
  ]
}

export const revalidate = 3600 // ISR: regenerate hourly
```

**Entry page** â€” add `generateStaticParams`:
```tsx
export async function generateStaticParams() {
  const payload = await getPayloadInstance()
  const collections = ['bosses', 'characters', 'relics', 'locations', 'minibosses']
  const params = []

  for (const collection of collections) {
    const result = await payload.find({
      collection: collection as any,
      limit: 100,
      select: { slug: true },
    })
    for (const doc of result.docs) {
      params.push({ category: collection, slug: (doc as any).slug })
    }
  }
  return params
}

export const revalidate = 3600
```

**Homepage** â€” add ISR:
```tsx
export const revalidate = 3600
```

### Key Notes
- `generateStaticParams` runs at build time only, not during ISR revalidation. New entries added after build will be handled on-demand (dynamic rendering) if `dynamicParams` defaults to `true`.
- In Next.js 16, `params` is a Promise and must be awaited â€” the codebase already does this correctly.

## Risks & Caveats

1. **DB URL migration**: Changing `staticDir` doesn't retroactively update existing media records' `url` field. Must re-seed.
2. **Ephemeral filesystems**: If deploying to Vercel/serverless, committing images to `public/media/` (D-02) is the correct approach. A cloud storage plugin (S3, Vercel Blob) would be better long-term but is out of scope.
3. **Typography plugin**: The existing `prose` CSS class does nothing without `@tailwindcss/typography`. Must install or use manual styles.
4. **Build time**: `generateStaticParams` fetching all entries at build time adds build duration. With ~20 entries total, this is negligible.
5. **LenisProvider**: The smooth-scroll provider wraps all content â€” the footer must stay inside it to avoid scroll behavior issues.
